# RhythmMotionCV 技术路线规划书

## 1. 项目定位
- 项目名称: RhythmMotionCV - 节奏运动跟随小游戏
- 课程属性: 纯计算机视觉应用 (不使用AIGC生成模型)
- 输出形式: Jupyter Notebook (`ipynb`) + 可复现实验结果
- 核心目标: 利用人体关键点检测与时序判定，实现“按节奏做动作”的实时交互游戏

## 2. 需求拆解
- 功能需求
  - 摄像头实时读取视频流
  - 姿态检测并提取关键点 (肩、腕、髋、鼻等)
  - 生成节奏序列 (BPM驱动)
  - 给出动作指令 (举手/下蹲/左移/右移)
  - 在节拍窗口内完成动作判定并计分
  - 可视化反馈 (动作提示、命中等级、进度条、倒计时)
- 非功能需求
  - 单机本地运行，无网络依赖
  - 普通笔记本摄像头可演示
  - 实时性目标: >= 15 FPS
  - 鲁棒性目标: 短时关键点丢失时不崩溃

## 3. 技术路线总览
1. 视觉输入层
- OpenCV读取摄像头帧，进行翻转和尺寸统一。

2. 关键点估计层
- MediaPipe Pose输出33个关键点。
- 仅选取动作判定所需关键点，降低规则复杂度。

3. 动作判定层 (规则法)
- 动作集合:
  - HandsUp: 双腕高于肩线
  - Squat: 髋部相对基线下移
  - MoveLeft / MoveRight: 身体中心相对基线水平偏移
- 基线自适应:
  - 游戏开始前3秒静态校准
  - 记录center_x、hip_y、shoulder_y中位数作为个人基线

4. 节奏引擎层
- 设定BPM -> 计算拍点间隔 (`60/BPM`)。
- 每个拍点绑定动作标签。
- 判定窗口:
  - Perfect: |t - beat_t| <= 0.12s
  - Good: 0.12s < |t - beat_t| <= 0.30s
  - Miss: 超过窗口仍未完成

5. 交互可视化层
- 实时骨架叠加
- 当前动作提示卡片
- 节奏进度条 + 下一拍倒计时
- 命中结果短暂弹窗 (Perfect/Good/Miss)
- 分数、连击、生命值、剩余时间

6. 数据记录与评估层
- 记录每拍判定结果、系统延迟估计、FPS统计
- 输出:
  - Accuracy = (Perfect + Good) / 总拍数
  - Perfect率
  - 平均FPS
  - 各动作类别命中率

## 4. 系统模块设计

### 4.1 PoseTracker
- 输入: BGR帧
- 输出: `PoseSnapshot`
  - `visible`
  - `shoulder_y`, `wrist_l_y`, `wrist_r_y`
  - `hip_y`, `center_x`
- 作用: CV感知入口，提供稳定姿态特征

### 4.2 BaselineCalibrator
- 输入: 连续`PoseSnapshot`
- 输出: `baseline_center_x`, `baseline_hip_y`, `baseline_shoulder_y`
- 策略: 校准阶段取中位数，降低偶发噪声影响

### 4.3 ActionJudge
- 输入: 当前快照 + 基线 + 动作标签
- 输出: bool(动作成立)
- 规则阈值:
  - 手臂上举阈值 `hands_margin`
  - 下蹲阈值 `squat_delta`
  - 横移阈值 `move_delta`

### 4.4 RhythmEngine
- 输入: BPM、总时长、动作列表
- 输出: 当前拍、下一拍、动作目标
- 功能:
  - 生成拍点序列
  - 节拍时间窗口判定
  - 统计Perfect/Good/Miss

### 4.5 GameController
- 状态机:
  - `calibrating -> playing -> game_over`
- 逻辑:
  - 指令下发、计分、连击、生命管理
  - 难度控制 (BPM递增或窗口收紧)

### 4.6 UIRenderer
- 使用OpenCV绘制HUD
- 统一视觉语言:
  - 绿色=成功
  - 黄色=普通命中
  - 红色=失败/警告

## 5. 关键算法说明

### 5.1 动作鲁棒判定
- 使用“连续帧命中”策略替代单帧触发。
- 例: 连续 >= N帧动作成立才判定命中，减少抖动误判。

### 5.2 时序评分模型
- 拍点`beat_t`附近依据时间误差分级:
  - `dt = abs(now - beat_t)`
  - `dt <= perfect_window` -> Perfect
  - `dt <= good_window` -> Good
  - 否则Miss

### 5.3 实时性能优化
- 仅保留单人检测
- 控制渲染分辨率 (如960x540)
- 逻辑与绘制轻量化，避免重型后处理

## 9. 风险与应对
- 风险: 光照差导致关键点不稳定
  - 应对: 提示用户站位，增加`visible`校验与平滑
- 风险: 摄像头帧率低
  - 应对: 降低分辨率，减少绘制开销
- 风险: 个体体型差异导致阈值不通用
  - 应对: 使用“相对基线”而非绝对像素阈值

- 本项目完全贴合CV课程: 视觉感知 + 规则推理 + 实时交互系统
